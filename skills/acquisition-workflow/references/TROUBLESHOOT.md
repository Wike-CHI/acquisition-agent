### ❌ 错误8：开发信 CTA 太通用

| 错误 | 正确 |
|------|------|
| CTA："Let's discuss this opportunity" | **CTA 必须具体可行**："Can I send you a product catalog for your next Overland Conveyor project?" |

> 来源：美国市场获客案例（2026-04-01）。5 封开发信全部 10/10 分的经验表明，具体 CTA 明确提升回复率。

### ❌ 错误9：差异化优势不量化

| 错误 | 正确 |
|------|------|
| "high quality vulcanizing equipment" | **量化技术参数**：\"Titanium heating plates with precise temperature control (±1°C)\" |

> 来源：美国市场获客案例（2026-04-01）。量化优势（±1°C、30%、40%、7-10 天）显著提升可信度。

### ❌ 错误10：开发信只有优点，没有痛点共情

| 错误 | 正确 |
|------|------|
| "Our equipment offers [优点1], [优点2], [优点3]" | **痛点 + 解决方案**："With [客户痛点], our [产品] can help you [解决痛点] by [我们的优势]" |

> 来源：业务员反馈（2026-04-01）。"太机械，只有优点" → 必须提到客户面临的挑战/痛点，才能建立共情。

**示例**：
- ❌ "Our vulcanizers offer titanium heating plates and 30% energy efficiency."
- ✅ "With rising energy costs in mining operations, our titanium heating plates help reduce energy consumption by 30% while maintaining precise temperature control."

## 防卡顿机制（v2.2.0）

### 流程卡顿风险分析

**高风险卡顿点（P0）**：
- **Step 1**：客户发现渠道全部失败 → 无线索可继续
- **Step 1.5**：无邮箱但继续流程 → 浪费资源
- **Step 4**：LinkedIn决策人搜索归因错误 → 找到错误的工具
- **Step 6**：开发信评分虚高跳过润色 → 质量不达标
- **Step 7.5**：邮件发送失败无重试 → 线索丢失

**中风险卡顿点（P1）**：
- **Step 2**：客户查重缓存不可用 → 可能重复开发
- **Step 3**：企业背调信息不足 → ICP评分不准确
- **Step 6**：评分循环（评分始终<9.0） → 无限循环
- **Step 8**：Pipeline更新文件锁定 → 数据写入失败

**低风险卡顿点（P2）**：
- **Step 5**：竞品分析信息缺失 → 差异化话术不够强
- **Step 9**：日报生成失败 → 无影响主要流程

### 8大防卡顿机制

#### 机制1：Skills Router（技能路由器）

**功能**：自动故障切换，根据任务类型智能调度技能

**优先级降级策略**：
```
优先级5（首选） → 优先级3（备选） → 优先级1（降级）
```

**实际应用（客户发现）**：
1. Exa搜索（首选）→ 失败
2. Google搜索（备选）→ 失败
3. 手动建议（降级）→ "请提供更多关键词或更换渠道"

**效果**：渠道失败自动降级，无需人工干预

---

#### 机制2：铁律：无邮箱不继续

**功能**：Step 1.5 强制检查，无邮箱停止流程

**实现逻辑**：
```python
if not email:
    logger.error("⚠️ 无邮箱，流程停止")
    return {
        "status": "blocked",
        "reason": "无邮箱",
        "suggestion": "请先获取联系方式（官网/LinkedIn/社媒）"
    }
```

**效果**：避免浪费资源在无效客户上

**功能**：Step 4 确保使用正确的工具搜索LinkedIn决策人

**错误模式**（已纠正）：
- ❌ LinkedIn决策人 → linkedin技能/MCP
- ✅ LinkedIn决策人 → **Exa搜索**

mcporter call "exa.web_search_exa(query: 'linkedin profile gerente compras banda transportadora Colombia', numResults: 8, includeDomains: ['linkedin.com'])"

**关键词模板**：西语/葡语 + 岗位 + 产品 + 国家

**效果**：工具调用正确，避免归因错误

#### 机制4：开发信生成 v2.0

**功能**：Step 6 强制润色 + 评分≥9.0分

**标准打磨流程**：
```
开发信生成
  ↓
第1轮润色（humanize-ai-text 自动检测+修复）
  ↓
第1轮评分（满分10分）
  ↓
评分 < 9.0分？
  ├─ 是 → 第2轮润色 → 第2轮评分
  └─ 否 → 最终版本（≥9.0分）
```

**评分标准**（满分10分）：
- 个性化程度（2.0分）
- 相关性（2.0分）
- 简洁性（2.0分）
- 语法质量（2.0分）
- 去AI味（2.0分）⭐ 强制调用 humanize-ai-text
- CTA有效性（2.0分）

**防循环机制**：最多2轮润色，避免无限循环

**效果**：质量保障，评分≥9.0分才通过

#### 机制5：邮件发送重试队列

**功能**：Step 7.5 邮件发送失败自动重试

**重试策略**：
```
发送失败 → 加入重试队列
  ↓
1小时后重试 → 失败
  ↓
2小时后重试 → 失败
  ↓
4小时后重试 → 失败
  ↓
通知用户：发送失败，手动处理
```

**实现逻辑**：
```python
retry_queue = [
    {"email": "test@example.com", "attempt": 0, "next_retry": "2026-04-01T15:45:00"},
]
# 每5分钟检查一次队列，自动发送重试
```

**效果**：线索不丢失，自动重试

#### 机制6：ContextManager（上下文管理器）

**功能**：防止上下文溢出，自动压缩和锚定

**压缩策略**：
- Token ≥ 65% 触发主动摘要
- 保留关键信息：TaskStateAnchor、客户信息、流程状态
- 删除冗余信息：中间步骤日志、临时变量

**锚定机制**：
```javascript
const anchor = {
    taskId: "task-001",
    goal: "为 FLSmidth 生成并发送开发信",
    target: "FLSmidth (A级客户，ICP 88/100)",
    constraints: ["必须评分≥9.0分", "使用 Exa 搜索 LinkedIn"],
    currentStep: "Step 6: 开发信生成"
};
// 上下文溢出后自动恢复
```

**效果**：上下文使用率 110% → 35%

#### 机制7：ECL 执行控制层

**功能**：执行轨迹追踪 + 循环检测 + 置信度评估 + 状态变化监控

**置信度评估**：
```
置信度 = 信息完整性 × 相关性 × 时效性
  ↓
置信度 < 0.6 → 补充缺失字段 → 人工介入
```

**循环检测**：
```
同一客户同一操作 ≥ 3次/天 → 触发警报
  ↓
查看执行轨迹 → 评估规则 → 人工决策
```

**状态变化监控**：
```
状态变化率 < 40% → 流程卡住
  ↓
分析停滞原因 → 优化流程
```

**效果**：任务成功率 50% → 95%

---

#### 机制8：4层抗遗忘系统

**功能**：L1/L2/L3/L4 记忆，防止信息丢失

**L1: MemOS 结构化内存**
- 每轮自动提取/注入
- 格式：`Task, Goal, Context, Actions, Results, Next`

**L2: 主动摘要压缩**
- Token ≥ 65% 触发
- 保留关键信息，删除冗余

**L3: ChromaDB 向量存储**
- 每轮存储 + 语义检索
- 目录：`C:\Users\Administrator\chromadb\honglong-acquisition`

**L4: CRM 每日快照**
- 每天 12:00 灾难恢复
- 备份所有客户信息

**效果**：信息丢失率 30% → 0%

## 回退方案

### 回退A：客户发现失败
```
1. 记录失败原因
2. 尝试其他关键词（变体、同义词）
3. 切换其他渠道
4. 所有渠道都失败 → 建议用户更换条件，保存失败记录
```

### 回退B：客户查重错误
```
1. 尝试本地缓存查重
2. 缓存不可用 → 跳过查重，标记"未查重"，人工确认时提醒
```

### 回退C：企业背调失败
```
1. 尝试其他背调渠道
2. 所有渠道都失败 → 标记"信息不足"，初步评分，建议人工补充
```

### 回退D：LinkedIn决策人渠道不可用
```
1. 自动切换到 Exa 搜索 LinkedIn 索引（已集成到 Skills Router）
2. Exa也失败 → Jina Reader 抓取 LinkedIn 公开页面
3. 都失败 → 标记"需人工查找"
```

> 注意：LinkedIn决策人搜索默认使用 Exa（不是 LinkedIn MCP），此回退仅作为备用方案。

### 回退E：邮件发送失败
```
1. 尝试其他邮箱
2. 无其他邮箱 → 标记"发送失败"，重试队列（1h→2h→4h），通知用户
```

## L3/L4 抗遗忘系统错误处理（v2.2.0 更新）

### ChromaDB 连接失败（L3）

```
症状: chroma:store 或 chroma:search 返回错误
原因: ChromaDB 服务未启动或路径不存在
处理:
  1. 检查目录: C:\Users\Administrator\chromadb\honglong-acquisition
  2. 启动 ChromaDB 服务
  3. 降级: 写入纯文本日志文件，标记"待同步"
  4. 通知: 记录 ChromaDB 连接异常
```

### 向量存储失败（L3）

```
症状: chroma:store 写入失败
处理:
  1. 重试 3 次（间隔 1 秒）
  2. 仍失败 → 写入备用文件: logs/chroma-fallback/{date}.json
  3. 在 CRM 中标记该客户"记忆待同步"
```

### 检索无结果（L3）

```
症状: chroma:search 返回空结果
处理:
  1. 返回"无历史记录"提示
  2. 标记为新客户或首次联系
  3. 建议: 从 Supermemory 补充相关信息
```

### Supermemory LanceDB 连接失败

```
症状: memory:add 或 memory:search 返回错误
处理:
  1. 检查目录: C:\Users\Administrator\lancedb\honglong
  2. 降级: 使用内存缓存
  3. 记录错误，待下次启动恢复
```

### 记忆标签错误（L3）

```
症状: 自动标签（has_quote/has_objection 等）识别错误
处理:
  1. 人工复核时纠正
  2. 将纠正结果记录到 Supermemory（effective_script）
  3. 下次相同场景优先使用纠正后的标签
```

### CRM 快照失败（L4）

```
症状: chroma:snapshot 在每天 12:00 执行失败
处理:
  1. 重试 3 次
  2. 仍失败 → 发送告警给业务员
  3. 手动补救: 手动运行 chroma:snapshot
  4. 检查 CRM 连接状态
```
