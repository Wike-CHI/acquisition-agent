# Security-Scan 代码安全审计插件

基于多 Agent 协同架构的代码安全审计插件，采用混合调度模式（独立 Agent + 流水线编排）执行系统化安全分析。核心引擎集成 **Grep 模式匹配** 与 **LSP 语义数据流追踪**，支持跨文件调用链分析、多态穿透、业务逻辑缺陷检测及跨仓库源码关联分析，并提供自动修复与 HTML 审计报告导出能力。

---

## 核心能力

| 维度 | 说明 |
|------|------|
| **自适应分级扫描** | 根据项目规模自动选择最优策略——Light（≤15 文件）/ Deep（>15），支持手动覆盖 |
| **复合指标判定** | 以文件数量为主轴，结合总代码行数修正与大文件预警，避免"文件少但代码量大"场景的误判 |
| **语义级深度分析** | 基于 LLM + LSP 追踪跨文件数据流，穿透多态与接口抽象层，检测传统 SAST 工具难以覆盖的复杂漏洞模式 |
| **多 Agent 协同** | 根据项目规模动态调度 1-6 个独立 Agent，三阶段流水线执行，兼顾扫描广度与分析深度 |
| **五步对抗验证** | 代码存在性校验 → 攻击链验证 → 红队对抗审查 → 审计质量评估 → 置信度评分，系统性消除误报 |
| **跨仓库源码关联** | 数据流追踪至第三方无源码模块时，引导用户关联外部源码仓库，规避依赖边界导致的漏报 |
| **大规模项目适配** | 自动执行模块分批、优先级排序与增量审计，通过 batch-worker 并行调度适配不同规模项目 |
| **量化置信度评分** | 0-100 分三维度独立评分体系：攻击链可达性（40 分）+ 防御措施（30 分）+ 数据源可控性（30 分） |
| **智能修复** | 对高置信度漏洞自动生成修复代码，经用户确认后应用，遵循最小变更与业务无损原则 |
| **增量 Diff 审计** | 纯配置 / 依赖变更快速通道 + 变更影响范围关联分析，显式追踪受影响的未变更文件 |
| **全中文交互** | 全流程中文进度提示与阶段性审计摘要输出 |
| **规则可扩展** | 支持自定义 Source / Sink / Sanitizer 规则，可通过新增 Agent 灵活扩展检测能力 |

## 分级扫描策略

根据项目规模自动选择最优扫描方案，支持用户通过 `--scan-level` 参数覆盖：

| 级别 | 适用场景 | 源文件数 | Agent 数量 | 预计耗时 | 核心特点 |
|------|---------|---------|-----------|---------|---------|
| **Light** | 小型项目 | <= 15 | 1（light-scan） | 约 15-20t | 单 Agent 完成全部工作，消除 Agent 间通信开销 |
| **Deep** | 中大型项目 | > 15 | 5-6 | 约 68-100t | 完整流水线，流式验证，分批策略 |

### 复合指标判定

不仅按文件数量分级，还引入总代码行数修正和大文件预警：

- 文件数 <= 15 但总行数 > 2000 --> 升级为 Deep
- 单文件 > 500 行 --> 标记大文件预警，优先深度分析

> 详细策略定义见 `references/strategies/scan-level-strategy.md`。

## 架构概览

### Deep 模式（>15 文件，完整流水线）

```
阶段 1（侦察）           阶段 2（扫描 + 流式验证）                    阶段 3（修复）
─────────────           ──────────────────────────                  ──────────
recon-lite (5t)         quick-scan ─────────────────┐              内联修复
    │                   recon-deep --> deep-scan ─────┤              + 报告生成
    └--> quick-scan 提前启动                          │
                        verification (流式验证) <──────┘
```

### Light 模式（<=15 文件，单 Agent）

```
阶段 1（编排器内联）     阶段 2（单 Agent）          阶段 3（修复）
──────────────         ──────────────────          ──────────
文件枚举 + 技术栈识别    light-scan (25t)           内联修复
LSP 探活 + 规则加载      ├ 模式匹配 (D1-D5)         + 报告生成
                        ├ LSP 数据流追踪
                        ├ 授权审计
                        ├ 内联反幻觉验证
                        └ 置信度评分
```

### Agent 目录

| # | Agent | 角色 | 阶段 | 关键能力 |
|---|-------|------|:----:|---------|
| 1 | recon-lite | 轻量侦察 — 文件枚举、技术栈识别、代码规模统计 | 1 | Glob + Bash（无 LSP 依赖） |
| 2 | recon-deep | 深度侦察 — 入口点枚举、端点-权限矩阵构建、攻击面映射、依赖解析 | 1 | LSP documentSymbol / workspaceSymbol + 依赖文件解析 |
| 3 | light-scan | 融合扫描 — Light 模式专用，集成模式匹配、LSP 数据流追踪与内联验证 | 2 | Grep + LSP + 内联验证（仅 Light 模式） |
| 4 | quick-scan | 广域扫描 — Grep 模式匹配、CVE 检测、配置安全基线扫描 | 2 | Grep 正则匹配 + 版本比对（无 LSP 依赖） |
| 5 | deep-scan | 语义扫描 — LSP 数据流追踪、授权覆盖审计、业务逻辑缺陷检测 | 2 | LSP 全套语义操作 |
| 6 | verification | 对抗验证 — 五步流水线：代码存在性校验 → 攻击链验证 → 红队对抗 → 质量评估 → 置信度评分 | 2（流式） | Glob/Read 事实校验 + LSP 路径验证 |
| +1 | batch-worker | 并行调度 — 大型项目分批并行执行 | 2 | 独立上下文窗口 |
| — | remediation | **内联执行** — 修复代码生成 + HTML 审计报告 | 3 | 编排器上下文内执行 |

## 操作指引

### 命令一览

| 命令 | 说明 | 适用场景 |
|------|------|---------|
| `/security-scan:project` | 全项目安全审计 | 新项目接入、定期安全审查、安全评估 |
| `/security-scan:diff` | Git 变更文件审计 | MR 审查、提交前检查、增量审计 |

### 使用示例

```bash
# 全项目审计（自动分级）
/security-scan:project

# 指定文件范围审计
/security-scan:project src/main/java --include *.java,*.kt --exclude *Test.java

# 微型项目强制深度扫描
/security-scan:project --scan-level deep

# 大型项目快速出结果
/security-scan:project --scan-level light

# 审计工作区全部变更
/security-scan:diff

# 审计指定 commit
/security-scan:diff --commit abc123

# 仅审计已暂存变更
/security-scan:diff --mode staged
```

### 环境变量

| 环境变量 | 说明 | 是否必须 |
|---------|------|:-------:|
| `YD_CODEBUDDY_REPORT_URL` | 审计结果上报地址 | 是 |
| `YD_CODEBUDDY_REPORT_TOKEN` | 上报认证 Token | 是 |

## 检测能力

### 置信度评分体系

置信度（0-100）基于三维度独立评分并累加：攻击链可达性（40 分）+ 防御措施（30 分）+ 数据源可控性（30 分）。高置信度门控（≥90）需同时满足 7 项前置条件，任一未通过则置信度上限限定为 89。

| 等级 | 分值范围 | 操作 |
|------|---------|------|
| 高 | >= 90 | 可自动修复 |
| 中 | 60-89 | 需人工审核 |
| 低 | < 60 | 仅供参考 |

> 详细评分维度和门控条件见 `ARCHITECTURE.md > 置信度评分体系`。

### 风险覆盖范围

插件内置 **8 大类、40 个标准风险类型**，用于统一风险归类、输出命名和报告统计。标准类型表位于 `resource/risk-type-taxonomy.yaml`，但它**不是扫描能力上限**：对于未预定义、但经代码语义分析确认的真实风险，Agent 会创建新的风险类型并正常输出，不会因为类型未登记而忽略问题。

| 大类 | 代表风险 | 典型场景 |
|-----|----------|---------|
| 注入类 | SQL 注入、命令注入、XSS、XXE、SSTI、表达式注入、JNDI 注入、代码执行 | 字符串拼接查询、模板渲染、表达式求值、危险执行接口 |
| 前端安全类 | 开放重定向、点击劫持、安全头缺失 | 未校验跳转地址、缺失 `X-Frame-Options` / CSP / HSTS |
| 认证与授权类 | 访问控制缺失、越权访问（IDOR）、认证绕过、CSRF | 接口未鉴权、资源归属未校验、可信来源绕过、状态变更接口缺少防护 |
| 数据与配置类 | 敏感信息泄露、硬编码凭证、弱加密、敏感端点暴露、不安全配置 | 日志泄露、源码密钥、弱算法、Actuator/Swagger 暴露、CORS/Debug 配置不当 |
| 服务端请求与数据处理类 | SSRF、路径遍历、不安全反序列化、不安全文件上传 | 外部 URL 请求、可控文件路径、危险对象恢复、上传校验缺失 |
| 依赖与供应链类 | 高危组件漏洞 | 第三方依赖存在已知 CVE 或高危版本 |
| 业务逻辑类 | 竞态条件、支付逻辑缺陷、幂等性缺失、数据隔离失效、回调信任缺失、事务完整性缺陷、频率限制缺失、状态机违规 | 重复支付、超卖、价格篡改、多租户越权、伪造回调、关键流程滥用 |
| 云安全类 | IMDS 无限制访问、IAM 过度授权、公开存储桶 | 云元数据可达、过宽权限策略、对象存储公开暴露 |

> 完整标准类型见 `resource/risk-type-taxonomy.yaml`。当发现 taxonomy 外的新风险类型时，结果会按新增 slug 生成对应 `finding-{slug}.json`。

### 检测方式

| 能力 | 说明 |
|------|------|
| 模式扫描 | `quick-scan` / `light-scan` 负责 Grep 模式匹配、Secrets 检测、配置基线扫描、依赖/CVE 初筛 |
| 语义数据流追踪 | `deep-scan` / `light-scan` 基于 LSP 追踪 Source 到 Sink 的跨文件调用链，穿透接口、多态和封装层 |
| 授权与业务逻辑分析 | 结合入口点、端点-权限矩阵和代码上下文，识别 IDOR、权限不一致、支付/回调/幂等/租户隔离等问题 |
| 五步对抗验证 | `verification` / `light-scan`（内联验证）对每个 finding 执行代码存在性校验、攻击链验证、对抗审查、全局审计质量评估、置信度评分 |
| 动态风险扩展 | 对标准类型表未覆盖、但代码中真实存在的风险，自动创建新类型并进入最终报告 |

### 支持的代码语言

| 语言 | LSP Server |
|------|-----------|
| Java / Kotlin | jdtls |
| JavaScript / TypeScript | typescript-language-server |
| Go | gopls |
| Python | pylsp |
| Rust | rust-analyzer |
| C# | omnisharp |
| Swift | sourcekit-lsp |

> LSP 提供精确的语义分析能力（调用链追踪、符号定位、类型推断）。当 LSP 不可用时，所有 agent 自动降级为 Grep + Read 模式，审计仍然有效但精度降低。
